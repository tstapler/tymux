# Pre-mortem: v1-release
**Date**: 2026-07-14

## Failure Modes

| # | Failure | First Symptom | Prevention | Severity |
|---|---------|--------------|------------|----------|
| 1 | Solo-dev, open-ended-timeline project stalls after Epic 3 (splits + breaking proto change) lands but before Epic 5 (keybindings/detach) or Epic 4/6 ship — leaving `master` with multi-window addressing complexity and no detach key or pane-focus-cycling, objectively worse day-to-day UX than the pre-1.0 single-pane MVP. | Developer catches themselves manually running `tymux split`/`tymux attach session:window.pane` from a second terminal to "switch panes" because there's no keybinding yet, and quietly goes back to using tmux day-to-day. | Pull a minimal Detach + NextWindow/PrevWindow keybinding slice out of Epic 5 into a small milestone that ships with or immediately after Epic 3, rather than strictly after Epic 4; at each alpha-tag checkpoint explicitly ask "is this more usable day-to-day than the pre-1.0 MVP?" and reprioritize if not. | P1 |
| 2 | Concurrent/overlapping window-resize recompute triggers for the *same* window (rapid drag-resize, or two clients resizing near-simultaneously) aren't serialized against each other, so a stale computation's unlocked `Pane::resize()` calls can commit after a newer one's — a transient torn-geometry state one layer above what ADR-004/Story 3.4 AC2 already guarantees within a single trigger. | A split pane's borders/dimensions briefly show inconsistent sizes between siblings during interactive resizing, then self-correct on the next resize event — easy to dismiss as an unreproducible one-off glitch. | Add a concrete Epic 3 task implementing either a true per-window serial recompute task/actor, or a generation counter checked at the "re-acquire to commit" step so superseded computations are discarded; add a regression test that fires two overlapping resize triggers for one window and asserts only the newer geometry is ever visible. | P3 |
| 3 | Story 1.4's musl spike runs once, early, on a throwaway crate that gets deleted, and Story 1.2's ongoing CI matrix never includes a musl build target — so nothing continuously verifies musl compatibility as Epics 4/5 add new dependencies (serde_json, vt100 0.16.2, toml/dirs), leaving Epic 8's real release cut as the only remaining point that could fail. | Epic 8's release workflow fails to cross-compile/statically-link against `x86_64-unknown-linux-musl` when cutting the real `v1.0.0` tag — the exact "last mile" failure the spike existed to prevent, just deferred. | Fold a lightweight `cargo build --workspace --target x86_64-unknown-linux-musl` (build-only) step into Story 1.2's CI matrix from Epic 1 onward, not just the one-time throwaway spike. | P2 |
| 4 | Epic 3 Story 3.2's property-based `LayoutNode` invariant suite — the plan's own named highest-correctness-risk item — is implemented by an agent as technically-present but shallow (narrow proptest strategies, rarely generating deep/adversarial trees), satisfying the literal merge gate without actually catching the bugs it's meant to. | A real `LayoutNode` bug (ratio drift, wrong collapse order on deep nesting) ships past Epic 3's gate and surfaces later during dogfooding as visibly corrupted/overlapping pane geometry in a real multi-split session. | Turn `build-vs-buy.md` §8's "human/reviewer pass, not agent-only" recommendation into a checkable gate in `validation.md`: require minimum generated tree depth, explicit coverage of the named nested-collapse/odd-remainder cases, and a mutation-testing pass (deliberately break the ratio-sum invariant and confirm the suite fails) before Epic 3 can merge. | P2 |
| 5 | Story 7.2 (unary `ListSessions`) explicitly wires its smoke test into CI as an ongoing job, but Story 7.3 (`Attach` bidi-stream + `CapturePane` — the actual point of Epic 7 and requirements.md Success Metric #3) has no equivalent "wire into CI" task, so nothing continuously re-verifies the TS `Attach` path once later epics touch the proto/engine/RPC teardown semantics. | Months after Epic 7 ships, an unrelated later change silently breaks the TS `Attach` example, discovered (if ever) only by someone manually re-running `clients/ts/examples/attach.ts`, not by CI on the PR that caused it. | Add an explicit task to Story 7.3 mirroring Story 7.2 Task 3 — wire the `Attach`+`CapturePane` integration test into CI as a standing job on every subsequent PR — and add it as a named line item in `validation.md`'s coverage map for Success Metric #3. | P2 |

## P1 Items (address before implementation)
- [ ] Failure #1 — Add a mini-milestone (e.g. "Epic 3.5") pulling Detach + NextWindow/PrevWindow keybindings out of Epic 5 to ship with/immediately after Epic 3, and add an explicit "still more usable than pre-1.0 MVP?" checkpoint question to each alpha-tag gate in plan.md §5 (Risk Control).
